# Contributing Guidelines

Thank you for contributing to this project! Please read and follow these guidelines to ensure a smooth and consistent workflow.

---

## 📌 Workflow
1. **Issues First**  
   - Before starting any work, create an issue or pick an existing one.  
   - Clearly describe the feature, fix, or improvement you plan to work on.

2. **Branch Naming**  
   - Create a branch from `main` using this format:  
     ```
     contributor-shortname/feature-name
     ```
     Example: `john/feature-authentication`

3. **Development Workflow**  
   - Work on your branch, following coding standards and conventions.  
   - Commit your changes with [Conventional Commits](https://www.conventionalcommits.org/) (see below).  
   - Push your branch and open a Pull Request (PR) to `main`.  
   - At least **1 developer review/approval** is required before merging.  
   - After merging, **delete the branch**.

---

## 📌 Coding Standards
- **JavaScript** → ESLint + Prettier enforced.  
---

## 📌 Commit Message Convention
We strictly follow **Conventional Commits**. Example formats:  

- `feat: add user authentication`  
- `fix: resolve login bug`  
- `update: improve dashboard layout`  
- `refactor: optimize database queries`  
- `docs: update contributing guidelines`  
- `chore(deps): update dependencies`

---

## 📌 Testing
- Contributors are encouraged to write **unit tests** or test manually.  
- Every PR must have at least **1 approval before merge**.  

---

## 📌 General Rules
- This project is private and proprietary to **Kudagon**.  
- Contributors must not redistribute or tamper with the project outside of assigned tasks.  
- All contributions should respect the repository's coding standards and workflow.  

---

💡 By contributing, you agree to follow these guidelines.
